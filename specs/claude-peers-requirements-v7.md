# claude-peers 要件定義書 v7.0
マルチエージェント協調システム

---

## 前提条件

- Claude Code v2.1.80 以降が必要
- claude.ai ログインが必須（APIキー認証は不可）
- Bun ランタイムが必要（チャネルサーバー実行用）
- 起動フラグ：`--dangerously-load-development-channels server:<name>`
- ⚠ リサーチプレビュー段階のため、プロトコル変更の可能性あり

---

## 1. プロジェクト概要

claude-peers は、複数の Claude Code セッションがローカルネットワーク上で相互に通信・協調し、大規模なソフトウェア開発タスクを自律的に分担・実行するマルチエージェントシステムです。

Claude Code の公式チャネルプロトコル（`notifications/claude/channel`）を活用し、セッション間のリアルタイムプッシュ通信を実現します。すべての処理はユーザーのマシン上で完結し、クラウドへのデータ送信は行いません。

### 設計方針

- 公式チャネルプロトコル活用：`notifications/claude/channel` でリアルタイムプッシュ
- 完全ローカル動作：クラウド不使用、低レイテンシー
- スケーラブル：セッション数に上限なし
- 自律性調整可能：MANUAL / HYBRID / FULL_AUTO の3モード切替

### 1.2 システムプロンプト管理

Dispatcher・Worker のシステムプロンプトはプロジェクトの CLAUDE.md を汚さず、claude-peers 専用ディレクトリで管理します。

| ファイル | 用途 |
|---|---|
| `~/.claude-peers/dispatcher.md` | Dispatcher用システムプロンプト |
| `~/.claude-peers/worker.md` | Worker用システムプロンプト |

起動時にこれらを自動で読み込みセッションに注入します。プロジェクトの CLAUDE.md は一切変更しません。

### 1.3 起動・画面管理

推奨環境は Windows です。Windows Terminal の複数タブで各セッションを管理します。tmux による分割表示はオプションです。

| 環境 | 画面管理方法 | 推奨 |
|---|---|---|
| Windows | Windows Terminal（複数タブ） + start-peers.ps1 | ✅ 推奨 |
| Windows | tmux（WSL2内での分割表示） | オプション |
| Mac | tmux（分割表示） | オプション |

**推奨運用**
- Dispatcher タブだけ見ておけば全体の進捗は把握できる
- 他の Worker タブは裏で動いており、切り替えれば確認可能
- 全セッションを同時に見たい場合のみ tmux を使用

## 2. システム全体像

### 2.1 アーキテクチャ

システムは Dispatcher と Worker の2種類のロールで構成されます。各セッションはチャネルサーバーを持ち、ブローカー経由でリアルタイムにメッセージを受け取ります。

| コンポーネント | 役割 | 数 |
|---|---|---|
| Dispatcher Claude | タスク分解・Worker管理・進捗監視・人間への報告 | 1体 |
| Worker Claude | Dispatcher からアサインされたタスクを実行 | N体（動的） |
| チャネルサーバー | 公式チャネルプロトコルで Claude とブローカーをつなぐ（Bun + TypeScript） | セッションごと |
| ローカルブローカー | セッション管理・メッセージルーティング・ファイルロック管理（Python / FastAPI） | 1プロセス |
| SQLite DB | ピア情報・メッセージ履歴・ロック状態の永続化 | 1ファイル |

### 2.2 通信フロー

公式チャネルプロトコルを使った通信フローです。

1. 人間が Dispatcher にゴールを伝える
2. Dispatcher がタスクを Worker にアサイン（チャネルサーバー経由でプッシュ）
3. Worker は reply ツールでブローカーに HTTP POST してメッセージを送信
4. ブローカーが宛先セッションのチャネルサーバーに HTTP POST でルーティング
5. チャネルサーバーが `notifications/claude/channel` で Claude にプッシュ
6. Dispatcher が進捗を集約して人間に報告

**v1.0からの変更点**
- 変更前：Claude ←カスタムMCPツール→ ブローカー（ポーリング）
- 変更後：Claude ←チャネルサーバー← ブローカー（SSEプッシュ、公式プロトコル）
- → プッシュ型になりレイテンシーが大幅改善。実装量も削減。

## 3. 機能要件

### 3.1 ピア管理

すべての Claude セッションをブローカーが一元管理します。

| 機能 | 詳細 |
|---|---|
| 自動登録 | セッション起動時にブローカーへ自動登録（session_id, 作業dir, gitリポジトリ, ロール） |
| ピア発見 | list_peers コマンドで全アクティブセッションを取得 |
| ハートビート | 30秒ごとにブローカーへ生存通知を送信。2回連続失敗（約60秒無応答）で落ちたと判定し自動削除 |
| メタデータ共有 | 作業ディレクトリ・現在のタスク・アクティブファイル・ロール情報を公開 |
| ロール自己宣言 | Claude が自身のロールを宣言（参考情報として管理、強制はしない） |

### 3.2 メッセージング

公式チャネルプロトコルを使ったメッセージングです。Claude からは MCP の reply ツール経由、Claude へは SSE プッシュで届きます。

| コマンド / イベント | 方向 | 説明 |
|---|---|---|
| reply（MCPツール） | Claude → ブローカー | 指定セッションへメッセージを送信 |
| broadcast（MCPツール） | Claude → 全セッション | 全アクティブセッションへ一斉送信 |
| notifications/claude/channel | ブローカー → Claude | メッセージをリアルタイムプッシュ（公式プロトコル） |
| check_messages（MCPツール） | Claude → ブローカー | 未読メッセージを取得（手動フォールバック） |
| set_summary（MCPツール） | Claude → ブローカー | 自セッションのタスク概要を更新 |

すべてのメッセージは SQLite に永続化されます。セッション再起動時のコンテキスト復元、デバッグ、ログ分析に使用します。

### 3.3 タスク管理

#### タスク分解ルール（Dispatcherに課す制約）

- 1つのタスク = 関連する一連のファイル群（依存関係のあるファイルは同一 Worker が担当）
- 同時ロックは1ファイルのみ。複数ファイルは直列（1つずつロック→編集→解放）で処理
- 依存関係のないタスクは別 Worker に分けて並列アサイン
- 同一ファイルを触るタスクは直列、異なるファイルは並列でアサイン
- このルールによりコンフリクトが構造的に発生しない設計とする

#### タスク並列化の判断フロー

Dispatcher は以下の順序でタスクを評価します。

1. 分解可能か判定 → 不可なら Worker 1体に直接アサイン
2. 並列化可能か判定 → 不可なら複数 Worker で直列実行
3. 並列化可能なら複数 Worker に同時アサイン

### 3.4 ファイルロック

ロールとロックは完全に分離して管理します。ファイル競合の防止はシステム側が機械的に処理し、ロール宣言に依存しません。

| 項目 | 仕様 |
|---|---|
| ロック粒度 | ファイル単位。同時ロックは1ファイルのみ。複数ファイルは直列で処理するため拡張不要 |
| 排他制御 | ロック取得中は他セッションが同一ファイルを編集不可 |
| タイムアウト | 一定時間応答のないセッションのロックは自動解放 |
| 通知 | ロック競合時は競合セッションへ自動通知 |
| 解放 | タスク完了時またはセッション終了時に自動解放 |

### 3.5 ロール管理

ロールは Dispatcher がタスク内容から判断して決定します。事前の固定ロール一覧（B）を基準としつつ、必要に応じてカスタムロールも追加できます（C）。

#### 基本ロール（デフォルト定義）

| ロール | 主な担当 |
|---|---|
| Backend | APIサーバー・DB・ビジネスロジック |
| Frontend | UI・状態管理・スタイリング |
| Tester | ユニットテスト・E2Eテスト・テストデータ |
| Researcher | 調査・ドキュメント収集・技術選定 |
| Reviewer | コードレビュー・品質チェック |
| DevOps | CI/CD・インフラ・デプロイ設定 |

Dispatcher がタスクを分析し、必要なロールを判断して Worker をスポーン。上記にないロールが必要な場合はカスタムロールを動的に定義します。

## 4. 自律性モード

自律性は3段階のモードで切り替えが可能です。ランタイム中でも変更できます。

| モード | Dispatcherの動作 | 人間の関与 |
|---|---|---|
| MANUAL | 全ステップで人間の承認を待つ | 高（都度承認） |
| HYBRID | タスク分解提案→承認後に自動実行 | 中（分解時のみ） |
| FULL_AUTO | ゴールから最終結果まで完全自律 | 低（結果確認のみ） |

**推奨運用**
- 初回・重要タスク → HYBRID（分解ロジックを確認しながら進める）
- 慣れてきたら → FULL_AUTO（ゴールだけ伝えてあとはお任せ）
- トラブル発生時 → MANUAL（ステップごとに確認しながらデバッグ）

## 5. チャネルサーバー仕様

各セッションに1つのチャネルサーバーが起動します。Bun + TypeScript で実装し、公式 MCP チャネルプロトコルに準拠します。

### 5.1 宣言する機能

| capability | 用途 |
|---|---|
| `claude/channel` | 必須。Claude へのプッシュ通知リスナーを登録 |
| `claude/channel/permission` | 権限リレー（ツール承認をリモートで可否判定） |
| `tools` | reply ツールなど Claude が呼び出すツールを公開 |

### 5.2 Claudeに公開するMCPツール一覧

| ツール名 | 引数 | 説明 |
|---|---|---|
| reply | peer_id, message | 指定セッションへメッセージ送信 |
| broadcast | message | 全セッションへ一斉送信 |
| check_messages | なし | 未読メッセージを取得（フォールバック） |
| set_summary | summary | 自セッションのタスク概要を更新 |
| lock_file | file_path | ファイルロックを取得 |
| unlock_file | file_path | ファイルロックを解放 |
| get_locks | なし | 現在のロック状態を取得 |
| set_role | role_name | 自セッションのロールを宣言 |
| set_mode | mode | 自律性モードを変更（MANUAL/HYBRID/FULL_AUTO） |
| list_peers | なし | 全アクティブセッションの一覧を取得 |

### 5.3 セッション起動コマンド

```bash
claude --dangerously-load-development-channels server:claude-peers
```

- `.mcp.json` にチャネルサーバーのエントリが必要
- Claude Code v2.1.80以降 + claude.ai ログインが必須

## 6. データ設計（SQLite）

| テーブル | 主なカラム | 用途 |
|---|---|---|
| peers | session_id, role, work_dir, git_repo, current_task, last_heartbeat | セッション管理 |
| messages | id, from_id, to_id, content, timestamp, read_flag | メッセージ履歴 |
| file_locks | file_path, session_id, acquired_at, expires_at | ロック管理 |
| tasks | task_id, assignee_id, status, parent_task_id, created_at | タスク追跡 |

## 7. 運用要件

### 7.1 障害対応

| ケース | 自動対応 | 人間の対応 |
|---|---|---|
| Worker が落ちた | Dispatcher が検知して通知 | 手動で再起動・タスク再アサインを判断 |
| ブローカーが落ちた | 自動再起動 + 全セッションに通知 | 再起動後に SQLite から状態を自動復元 |
| Dispatcher が落ちた | Worker は作業を継続 | 新しい Dispatcher を起動、SQLite から状態を復元して引き継ぎ |

### 7.2 システム設定値

| 設定項目 | 値 | 備考 |
|---|---|---|
| ブローカーポート | 7799 | localhost:7799 固定 |
| ハートビート間隔 | 30秒 | Worker が30秒ごとにブローカーへ生存通知 |
| タイムアウト判定 | 60秒（2回連続失敗） | 最後の生存通知から60秒経過で落ちたと判定 |
| デフォルトセッション数 | 4（Dispatcher×1 + Worker×3） | `start-peers.ps1 -workers N` で変更可能 |
| 進捗自動報告間隔 | 3分ごと | Dispatcher が人間に全 Worker の状況を報告 |

### 7.3 タイムアウト設定

Worker が無応答になった場合、2段階で対処します。自動での再アサインは行わず、人間が最終判断します。

| タイミング | 動作 |
|---|---|
| 5分無応答 | ログに記録のみ |
| 10分無応答 | 人間にアラート通知、以降は指示があるまで待機 |
| 人間の指示後 | 再アサイン／待機継続／キャンセルのいずれかを実行 |

**タイムアウト時の選択肢**
- 「再アサインして」→ タスクを別 Worker に移す
- 「もう少し待って」→ タイムアウトをリセットして継続監視
- 「キャンセルして」→ タスクを破棄

## 8. 非機能要件

| 項目 | 要件 |
|---|---|
| ローカル動作 | すべての処理をユーザーマシン上で完結。外部クラウド不使用 |
| レイテンシー | ピア間メッセージのラウンドトリップを100ms以内を目標 |
| スケーラビリティ | デフォルト4セッション（Dispatcher×1 + Worker×3）。`start-peers.ps1 -workers N` で変更可能 |
| 耐障害性 | セッション障害時の自動ロック解放・タスク再アサインをサポート |
| セキュリティ | localhost のみでリッスン。外部ネットワークへの公開は行わない |
| 永続性 | メッセージ・ロック・タスク履歴を SQLite に保存し再起動後も復元可能 |
| プロジェクト分離 | 起動時にプロジェクト名を namespace としてブローカーに渡す。異なるプロジェクト間の通信は発生しない |

| 項目 | 方針 |

## 10. UX要件

### 10.1 進捗の可視化

Dispatcher は2つの方法で進捗を報告します。

| 方法 | 動作 |
|---|---|
| 定期自動報告 | 3分ごとに Dispatcher が全 Worker の状況をまとめて報告 |
| オンデマンド報告 | 人間が「進捗は？」と聞いたときに即座に報告 |

### 10.2 ログ管理

すべてのログは `~/.claude-peers/logs/` に集約します。プロジェクトのリポジトリには一切混入しません。

| ファイル | 内容 |
|---|---|
| `~/.claude-peers/logs/broker.log` | ブローカーの動作ログ（起動・停止・ルーティング） |
| `~/.claude-peers/logs/sessions/session-xxx.log` | セッションごとのログ（メッセージ・タスク・エラー） |

**`~/.claude-peers/` の全体構造**

```
~/.claude-peers/
  dispatcher.md          # Dispatcher用システムプロンプト
  worker.md              # Worker用システムプロンプト
  logs/
    broker.log           # ブローカーログ
    sessions/            # セッションごとのログ
```

### 10.3 Dispatcher定期報告の内容

Dispatcher は3分ごとに以下の内容を人間に報告します。

| 報告項目 | 内容 |
|---|---|
| 各 Worker の状況 | ロール・現在のタスク・進捗状況 |
| タスク全体 | 完了タスク数・残りタスク数・全体の進捗率 |
| 異常検知 | エラーが出ている Worker の有無と内容 |

## 11. 実装フェーズ

| フェーズ | 内容 | 技術スタック | 優先度 |
|---|---|---|---|
| Phase 1 | ブローカーデーモン（ピア登録・ハートビート・メッセージルーティング） | Python / FastAPI + SQLite | 最高 |
| Phase 2 | チャネルサーバー（公式プロトコル対応・reply ツール・list_peers 等） | Bun + TypeScript + MCP SDK | 最高 |
| Phase 3 | ファイルロック機構 | ブローカー側に追加 | 高 |
| Phase 4 | Dispatcher ロジック（タスク分解・Worker スポーン） | システムプロンプト設計 | 高 |
| Phase 5 | 自律性モード切替（MANUAL/HYBRID/FULL_AUTO） | Dispatcher 制御ロジック | 中 |
| Phase 6 | ブロードキャスト・タスクキュー・カスタムロール | ブローカー拡張 | 低 |

**次のステップ**
- Phase 1：ブローカーデーモン実装（Python / FastAPI + SQLite）
- Phase 2：チャネルサーバー実装（Bun + TypeScript + @modelcontextprotocol/sdk）
- 動作確認：`--dangerously-load-development-channels` で2セッション間の通信テスト
